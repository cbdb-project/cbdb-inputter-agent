# CBDB Inputter Agent — Implementation Plan

Status: **all 7 milestones implemented and reviewed** (see §10 and
`docs/02-review-log.md`). This document is kept as the architectural record and
rationale, not as a pending TODO list. See `00-target-system-brief.md` for the
target-system facts this plan relies on (including facts confirmed live during
Milestone 7), and `03-extraction-review-workflow.md` for the source-text →
proposal → human-review pipeline that precedes submission for unstructured input.

## 1. Goal

Build a repo that acts as an **authorized, attributable API client** for
`cbdb-online-main-server`'s `/api/v2/*` Mutation API — simulating a human data-entry
user, but scripted, for bulk/repetitive biographical record entry. Every write must:

1. Be authenticated as a real CBDB user (via a Sanctum Personal Access Token supplied
   by the user, never generated by the agent).
2. Land only on endpoints confirmed to write to `audit_log` + `operations`
   transactionally (`/api/v2/create|mutate|delete`, `mode: "direct"` — see brief §3–4).
3. Be logged locally too (request/response, timing, outcome) so we have our own
   independent audit trail in addition to the server's, and can reconcile the two.

Non-goals: this repo does not modify `cbdb-online-main-server` itself, does not add
server-side audit logging (already exists — brief §4), and does not implement
`mode: "proposal"` / crowdsourcing review workflows.

## 3. Configuration — `.env`

Rationale: user explicitly wants the endpoint itself in `.env` so switching between a
local dev server and production is a one-line change, and wants `.env` gitignored with
a committed `.env.sample`.

`.env.sample`:

```dotenv
# Base URL of the cbdb-online-main-server instance to target.
# Production: https://input.cbdb.fas.harvard.edu
# Local dev:  http://localhost:8000 (Laravel's `php artisan serve` default — also
#             confirmed by the user as the actual port for the standing local
#             instance, see AGENTS.md's Local dev section). Always check the real
#             .env for the port actually in use rather than hardcoding this.
CBDB_API_BASE_URL=https://input.cbdb.fas.harvard.edu

# Sanctum Personal Access Token, created manually at {CBDB_API_BASE_URL}/profile.
# Never commit a real value here. The agent never creates or rotates this itself.
CBDB_API_TOKEN=

# Safety switch: when true, the client builds requests and logs them but does not
# send mutating calls (create/mutate/delete). GET calls still go through. Default true
# so a fresh checkout can never accidentally write to production.
CBDB_DRY_RUN=true

# Second safety gate: must equal the EXACT value of CBDB_API_BASE_URL above (not just
# "true") before any mutating call is sent, whenever CBDB_DRY_RUN=false. URL-pinned
# rather than boolean so changing CBDB_API_BASE_URL always re-locks this gate. See
# AGENTS.md.
CBDB_CONFIRM_PROD=

# Self-imposed client-side rate limit (requests per minute) — independent of whatever
# the server enforces. See docs/00-target-system-brief.md section 6.
CBDB_MAX_REQUESTS_PER_MINUTE=60

# Local audit log location (JSONL, one line per API call attempt).
CBDB_LOCAL_AUDIT_LOG_DIR=./logs
```

`config.py` responsibilities:
- Load `.env` via `python-dotenv`.
- Fail fast (raise before any HTTP call) if `CBDB_API_TOKEN` is empty and
  `CBDB_DRY_RUN` is not true.
- Require `CBDB_CONFIRM_PROD` to equal the exact current `CBDB_API_BASE_URL` string
  as a second gate whenever `CBDB_DRY_RUN=false`, for **any** target host. URL-pinning
  (not a plain boolean) means changing `CBDB_API_BASE_URL` — e.g. from a local dev
  server to production — automatically invalidates the previous confirmation, so a
  silent target switch can never inherit an earlier "yes, go live" decision.

## 4. Local audit logging (`audit_log.py`)

Independent from — and a safety net on top of — the server's own `audit_log` table
(brief §4). Purpose: even if the server call fails, times out ambiguously, or the
server-side log has a gap (brief's documented known gaps), we have our own record of
what the agent *attempted* and what it *observed*.

For every API call (not just mutations — reads too, for full traceability):
- timestamp (UTC, ISO8601)
- resource, operation, mode
- full request payload (as sent)
- full response payload (as received) or error
- HTTP status code
- the `operation_id` echoed back by the server, if present, so our local log can be
  cross-referenced against the server's `operations`/`audit_log` rows
- dry-run flag (was this actually sent or just simulated)
- a locally generated correlation ID (UUID4) independent of server `operation_id`,
  so we can trace a call even if the server never got it (network failure)

Format: append-only JSONL, one file per day (`logs/2026-07-08.jsonl`), never
overwritten, never deleted by the agent. This mirrors the server's append-only
`audit_log` design intentionally.

## 5. HTTP client (`http_client.py`)

- `requests.Session` with `Authorization: Bearer {token}` header set once.
- Wraps every call with: local audit logging (§4), client-side rate limiting
  (a fixed minimum-interval limiter at `CBDB_MAX_REQUESTS_PER_MINUTE` — simpler
  than a token bucket and deliberately more conservative: no burst allowance),
  and structured error mapping:
  - 401 → raise `AuthenticationError` (bad/expired token — do not retry)
  - 403 → raise `AuthorizationError` (account lacks `canWriteDirectly()` — do not retry)
  - 409/422 → raise `ConflictError` with the response body attached — surfaced to the
    human, never auto-retried with modified data (brief §3/§6)
  - 429 → exponential backoff retry (bounded, e.g. max 3 attempts)
  - 5xx → limited retry with backoff, then surface
- Dry-run mode: for mutating verbs, log the would-be request and return a synthetic
  "not sent" result instead of calling `requests`.

## 6. Mutation API wrappers (`mutation_api.py`, `models.py`)

- One typed function per resource + operation, e.g. `create_person(...)`,
  `create_address(...)`, `update_kinship(...)`, one per resource in
  `docs/04-field-whitelists.md` (`basicinformation`, `altnames`, `addresses`,
  `kinship`, `postings`/`offices`, `associations`, `entries`, `events`,
  `possessions`, `social_institutions`, `sources`, `statuses`, `texts`) — use each
  resource's *canonical* alias from that document, not the shorthand names in the
  original brief §3 sketch (e.g. `associations`, not `assoc`; `social_institutions`,
  not `socialinst`, for update calls specifically — see `04-field-whitelists.md` §12's
  alias gap).
- Each wrapper builds the JSON envelope from brief §3, always sets `mode: "direct"`,
  and validates the payload against a per-resource field whitelist declared in
  `models.py` **before** sending — client-side rejection of unknown fields, mirroring
  (and front-running) the server's own whitelist rejection, to fail fast with a clearer
  error message.
- `person_id.py`: before creating a person, call `GET /api/v2/get` /
  `GET /api/v2/persons` to find `max(c_personid)`, and validate any client-supplied ID
  against the `<= max + 10000` and not-already-taken rules from the brief, raising
  before ever sending a doomed request.
- Sub-resource creates require `person_id` to already exist — wrapper checks this
  (via `GET /api/v2/get`) and raises a clear local error instead of relying on the
  server's 4xx if the person doesn't exist yet.

## 7. CLI / entry point (`cli.py`, `batch_runner.py`) — as implemented

`python -m cbdb_agent submit --input data/inbox/records.json` (a JSON array of
records; see `staging.load_input_batch()`'s docstring for the exact shape) or
`--staging <path>` (a YAML file from the extraction workflow, §4 below):
- Both paths converge on one internal representation (`staging.StagingBatch`) and
  one execution engine (`batch_runner.run_batch()`), rather than duplicating
  submission logic per input format.
- Validates the whole batch upfront (`staging.validate_for_submit()`) before
  sending anything — a structural error blocks the entire batch, distinct from a
  runtime failure on one proposal (see below).
- Submits proposals in dependency order (person before sub-resources, §6),
  logging each API call locally (§4) regardless of outcome.
- On a runtime failure (conflict, permission error) for one proposal, that
  proposal is marked `failed` and any proposal depending on it is marked
  `skipped_dependency_failed` — the rest of the batch continues
  (`AGENTS.md` rule 5's isolation applies per-proposal, not just per-record).
- On any real (non-dry-run) submit attempt, the whole source file (input JSON or
  staging YAML) is archived to `data/processed/<batch_id>/` alongside a
  `results.json` with every proposal's outcome — a repeat submission of the same
  `batch_id` gets its own `-attempt2`/`-attempt3` directory rather than
  overwriting the previous attempt's results.
- `--dry-run` flag mirrors `CBDB_DRY_RUN` and can force it on even if `.env` disables
  it, but cannot force it off (safety only goes one direction from the CLI); a
  dry-run never archives anything, since nothing was actually attempted.

## 8. Skill (`skills/cbdb-data-entry/SKILL.md`)

A skill so a Claude Code session (this agent or a future one) can be asked "录入这批
数据到 CBDB" and it knows to:
1. Check `.env` is configured and `CBDB_DRY_RUN` state, confirm with the user before
   any non-dry-run production write.
2. Validate the input file shape before calling the CLI.
3. Run `python -m cbdb_agent submit ...` and surface the per-record summary.
4. Never construct raw HTTP calls itself, bypassing `mutation_api.py`'s validation and
   `audit_log.py`'s logging — always go through the CLI/library so the audit trail
   guarantee in §4 holds.

## 9. AGENTS.md (repo root)

Documents for any agent working in this repo:
- The hard rule: only ever call `/api/v2/create|mutate|delete|get` (`mode: "direct"`
  for writes), plus the read-only `/api/v2/persons`/`/api/v2/operations` list
  endpoints — never the legacy `/basicinformation/*` web routes, `v1` GET-based CRUD,
  or `Api\OperationsController` (brief §3–4, §6 — those are the paths with
  unconfirmed or documented audit gaps).
- Never commit `.env` or any real token.
- Never bypass `http_client.py`'s local audit logging for a "quick" script.
- Production writes require the explicit two-gate confirmation in §3.
- Where to find the target-system facts (`docs/00-target-system-brief.md`) and this
  plan (`docs/01-implementation-plan.md`), and that the brief is a snapshot that should
  be re-verified against the live `cbdb-online-main-server` repo if anything about auth
  or audit behavior seems off.

## 11. Review workflow (applies to every milestone above)

Per user's explicit process requirement:

1. After finishing a milestone's code, dispatch a **review agent** (Explore or
   general-purpose subagent) that reads the actual diff/new files and the relevant
   target-system brief sections, and reports findings.
2. Fix any issues found; repeat step 1 until the review agent reports no serious
   issues.
3. Then run **codex** as an independent second-opinion reviewer, non-interactively:
   ```powershell
   $env:HTTPS_PROXY = "http://127.0.0.1:7890"; $env:HTTP_PROXY = "http://127.0.0.1:7890"
   Write-Output "<review prompt describing the milestone's diff and asking for a code review, focused on correctness, security, and audit-trail integrity>" | codex exec --dangerously-bypass-approvals-and-sandbox
   ```
4. Fix any issues codex raises; re-run codex until it reports no serious issues.
5. Record the outcome of both review passes (agent + codex) in
   `docs/02-review-log.md` before starting the next milestone.
6. Only then move to the next milestone.
